Generally, this is what this talk will cover. I'm going to explain what's right about GeoJSON, why it's right and why it matters, and suggest how we can reform standardization and agree on more humanely-sized specs.

A Timeline

  • 1998: XML 1.0
  • 2002: D. Crockford launches json.org
  • 2005: AJAX
  • 2006: RFC 4627, "The application/json Media Type for JavaScript Object Notation (JSON)"
  • 2009: The Web stops calling XML back
I'm exaggerating a bit at the end, but around 2009 (maybe it was 2008) brand new services might for the first time offer JSON only, or XML as an afterthought. JSON document databases like CouchDB and MongoDB also played a role.

JSON's Design Goals

  • to be minimal
  • portable
  • textual
  • a subset of JavaScript
  • (and other languages with Map literals)
Text trades performance for transparency and hackability. While Crockford was targeting Javascript, other languages with Map literals are interchangeable with Javascript where JSON is concerned. Python's syntax is practically identical. JSON handles Replacing XML globally is not one of JSON's design goals.

JSON is self-describing

{"foo": 5, "bar": "5"}

vs.

55
The format is self-describing in a way that XML is not. Foo is a number. Bar is a string. No schema is needed to explain that.
